## 一、开场与项目定位（约 1 分钟）

老师好，我们小组本次汇报的项目是 **StudyPilot 智能学习助手**。

这个项目主要面向大学生，解决三个比较常见的问题。

第一，课程资料很多，但学生很难快速找到答案，更难确认答案到底来自哪一份课件。

第二，学生通常知道自己需要复习，却不知道应该先学什么、每天学多久，以及计划落后后应该如何继续安排。

第三，传统学习平台往往只记录资料和任务，大模型聊天工具又不能可靠保存用户的课程、计划、答题和掌握度数据，二者之间缺少一个持续更新的学习闭环。

因此，我们的目标并不是只做一个聊天机器人，而是做一个能够长期记录学习过程的智能学习平台。我们的核心闭环是：

> 用户创建课程并上传资料，系统处理资料并提供带引用的问答；再根据用户偏好、考试时间、计划任务、答题结果和知识点掌握度生成学习计划与推荐；用户完成任务和练习后，数据继续反馈到掌握度、统计和周报中。

目前项目已经从早期页面原型推进到了 **真实业务闭环和可见交互验收阶段**。当前开发分支是 `liverharden`，最新工作主要集中在交互可靠性、并发操作锁和自动化验收加固上。

## 二、项目总体架构（约 2 分钟）

从技术架构上看，我们采用前后端分离和容器化部署。

### 2.1 前端

前端使用：

- Vue 3；
- TypeScript；
- Vite；
- Element Plus；
- Pinia；
- Axios；
- ECharts；
- Playwright。

前端负责登录注册、课程管理、资料上传、智能问答、学习计划、今日任务、推荐、练习、错题本、统计、周报和日历等用户交互。

### 2.2 后端

后端使用 Python 和 FastAPI，配合 SQLAlchemy、Pydantic 和 JWT，负责：

- 用户认证和权限校验；
- 课程、文档、计划和任务等业务规则；
- RAG 问答；
- 推荐和统计；
- 数据一致性和幂等控制；
- 对前端提供统一 REST API。

### 2.3 数据层

数据层主要由三部分组成。

第一，**PostgreSQL 是权威业务数据库**，保存用户、课程、文档、计划版本、学习任务、答题记录、掌握度、推荐反馈、聊天记录、日历事件和异步任务结果。

第二，**pgvector 用于保存文档切块向量并支持语义检索**。它是 PostgreSQL 的向量扩展，因此检索时可以同时按照用户、课程、文档和版本做权限过滤，避免跨课程或跨用户检索。

第三，**Redis 用于 Celery 队列、短期状态和任务进度**。Redis 中的数据不是最终事实，出现不一致时以 PostgreSQL 为准。

### 2.4 异步任务与 MCP

对于文档解析、Embedding 和周报生成等耗时操作，我们使用 Celery Worker 异步执行，避免长任务阻塞 API 和页面。

项目还设计并实现了独立的 MCP Server，将受控业务能力封装成 18 个 MCP 工具。对于读取操作可以直接调用，对于创建、修改和删除等写操作，采用“预览—确认—执行”的两阶段确认方式。

需要特别说明：当前日历操作的是 StudyPilot 本地日历，Outlook 和 Google Calendar 还没有自动接入。

整个项目通过 Docker Compose 组织 PostgreSQL、Redis、FastAPI、Celery Worker 和前端 Nginx，可以在本地统一启动。

## 三、目前已经完成的核心功能（约 5 分钟）

### 3.1 用户认证和数据隔离

首先，我们已经完成了真实的注册、登录、JWT 会话恢复和退出登录闭环。

用户登录后，前端会通过 `/users/me` 恢复真实用户信息。真实模式下，如果后端认证失败，前端不会生成虚假的用户或者 Token。

在安全方面，我们不仅做了页面路由保护，还在后端对课程、文档、任务、错题、推荐和日历事件做资源归属校验。

自动化验收中，我们还使用同一个浏览器完成了 A 用户、B 用户、再切回 A 用户的完整流程，验证 B 用户无法看到 A 用户的课程、资料、任务、错题、推荐反馈、周报和日历事件，重新登录 A 后原有数据仍然存在。

另外，我们补充了全局 401 会话失效处理。多个业务请求同时返回 401 时，只会执行一次清理、提示和登录跳转；旧请求延迟返回的 401 也不会误伤用户的新会话。

### 3.2 课程与资料处理

课程模块已经完成：

- 课程创建；
- 列表查看；
- 课程详情；
- 编辑；
- 归档；
- URL 刷新恢复；
- 多课程上下文隔离。

资料模块已经接入真实文件上传，支持 TXT、Markdown 和 PDF 等类型的后端校验。上传后会创建真实文档记录和异步任务，页面根据真实的 `courseId`、`documentId` 和 `taskId` 展示处理状态。

我们还处理了：

- 空文件；
- 空文本；
- 不支持的扩展名；
- 文件过大；
- 重复点击上传；
- 失败后的重新解析；
- 错配 URL；
- 跨课程任务访问。

在 Docker 联调中，TXT 文件可以经过 Celery Worker 真实处理，最终生成文档页、文本块和成功任务状态，并且刷新页面后数据仍然存在。

### 3.3 带引用的 RAG 智能问答

智能问答已经接入真实课程、已就绪资料、会话列表、消息历史和引用数据。

用户提问时，后端会先校验用户是否有权限访问课程和资料，再从当前课程、当前文档版本的文本块中检索相关内容。返回结果中会包含资料名称、原文片段等引用信息。

这个设计和普通聊天机器人的区别在于：

- 问答范围受课程和文档约束；
- 引用只能来自本次检索到的证据集合；
- 证据不足时可以明确提示资料不足；
- 问题、回答和引用都会持久化；
- 刷新页面或者切换会话后仍然可以恢复历史。

目前问答交互已经通过桌面端和移动端浏览器验收，包括发送、刷新恢复、双会话切换、引用展示、无引用状态和请求失败后的重试。

但我们需要如实说明：**目前使用的是离线 Provider 验证完整业务链路，外部大模型 Chat 和 Embedding Provider 尚未正式接入，因此当前阶段不评价模型回答质量。**

### 3.4 学习计划和今日任务

学习计划不是让模型输出一段不可执行的自然语言，而是保存为数据库中的结构化计划版本和任务。

目前已经完成：

- 根据课程和学习偏好生成候选计划；
- 展示候选版本；
- 用户确认后将候选版本变为活动版本；
- 旧版本和候选版本的任务不会混入今日任务；
- 页面刷新后确认状态和当前计划仍可恢复。

今日任务完成时，会记录真实学习分钟数和 LearningRecord，并更新知识点掌握度。

对于重复点击、网络失败后重试和并发提交，我们做了幂等控制。后端通过行锁和唯一约束保证同一个任务只产生一条完成记录，掌握度也只更新一次。

需要说明的是，完整的自动延期、专注计时、计划历史和自动动态调整还属于后续能力；当前重点已经完成“计划生成—用户确认—任务执行—结果持久化”的基础闭环。

### 3.5 练习、错题本和掌握度

练习模块已经完成题目获取、答案提交、结果解析、耗时记录和下一题操作。

为了防止网络重试造成重复答题，我们为每次提交生成 `submission_id`。只有课程、题目、答案和耗时完全一致时才允许幂等重放；如果同一个提交 ID 被用于不同答案，后端会返回冲突错误。

答题后系统会：

- 保存 PracticeAttempt；
- 保存学习记录；
- 错题进入错题本；
- 更新知识点掌握度；
- 为推荐和统计提供真实行为依据。

错题本支持筛选、分页、查看详情、标记、移出和再次练习。掌握度页面会区分“存在真实答题证据”和“仅有初始数据”，不会把没有任何尝试的数据伪装成真实掌握情况。

### 3.6 可解释推荐、统计和周报

推荐中心不是随机生成一段 AI 文案，而是根据真实数据生成规则化推荐。目前包含：

- 学习任务；
- 薄弱点复习；
- 资料与问答；
- 学习计划；
- 学习复盘。

推荐会综合活动计划任务、掌握度、就绪资料、考试日期和学习记录，并给出推荐理由。用户的点击、有帮助和不感兴趣反馈会按真实推荐目标持久化。

最近一次加固中，我们解决了推荐卡片并发操作问题：当一张卡片的主操作正在等待反馈时，其他卡片的主操作也会被锁定，避免重复写入和重复跳转；各卡片的“有帮助”和“不感兴趣”反馈仍然保持独立状态。

统计模块已经完成：

- 7 天和 30 天范围；
- 当前周期与上一周期比较；
- 每日趋势；
- 课程分布；
- 49 天热力图；
- 连续学习天数；
- 练习正确率；
- CSV 导出。

统计严格按照用户时区计算，并且只聚合真实完成记录。CSV 使用 UTF-8 BOM，并加入了公式注入保护。

周报会根据持久化学习记录、当前活动计划和掌握度生成结构化结果，支持在任务中心查看，并可以导出 Markdown 文件。

### 3.7 长时任务和学习日历

长时任务中心已经支持文档解析任务和学习周报任务，包括：

- 列表和详情；
- 状态筛选；
- 分页；
- 自动刷新；
- 取消；
- 失败重试；
- URL 任务恢复。

学习日历已经完成本地事件创建、修改、删除、计划任务同步、周导航、时区处理和 ICS 导出。

日历写操作使用两阶段确认：先展示预览，再由用户明确确认，最后执行写入。我们也处理了幂等和并发写入，重复确认不会创建重复事件。

对于夏令时边界，我们专门增加了测试。例如纽约时区春季不存在的 `02:30` 会被拒绝，秋季重复时间会明确采用稳定的时间解释。

目前可以通过 ICS 手动导入第三方日历，但 Outlook 和 Google Calendar 自动同步尚未接入。

# 现场演示流程（8～10 分钟）

如果汇报后需要现场演示，建议不要随机点页面，而是按照一条完整学习故事展示。

## 一、演示前准备

提前执行：

```powershell
cd D:\SumMer_practice
docker compose up -d --build
docker compose ps
```

确认以下地址正常：

- 前端：<http://localhost:8080>
- 后端健康检查：<http://localhost:8000/health>
- API 文档：<http://localhost:8000/docs>

建议提前准备：

- 一个可登录的演示账号；
- 一门“数据库系统”课程；
- 一份较短的 TXT 或 Markdown 课程资料；
- 一个已经处理完成的文档；
- 一份已经确认的学习计划；
- 至少一道练习题和一条学习记录。

## 二、演示步骤与讲解词

### 第一步：登录和首页

**操作：** 登录并打开学习首页。

**讲解：**

> 这里展示的课程、今日任务、学习时长、掌握度和任务状态都来自当前用户的真实数据。真实模式下后端失败会展示错误，不会自动回退到演示数据。

### 第二步：课程与资料上传

**操作：** 进入课程详情，上传准备好的 TXT 文件。

**讲解：**

> 上传后系统创建真实文档和异步任务。文件解析由 Celery Worker 处理，前端根据真实任务 ID 展示状态。课程、文档和任务之间都有归属校验，替换 URL 中的 ID 不能访问其他课程或其他用户的数据。

如果现场网络或构建速度不稳定，可直接打开提前处理完成的资料，不要现场等待大 PDF。

### 第三步：带引用问答

**操作：** 进入智能问答，提问：

> 什么是第三范式？请根据课程资料回答。

**讲解：**

> 系统不是直接让模型自由回答，而是先在当前课程资料中检索，再把引用与答案绑定。刷新页面后会话、回答和引用仍然存在。当前演示使用离线 Provider 验证链路，因此我们重点展示可追溯性和业务闭环，不宣称外部模型质量已经完成验证。

### 第四步：生成并确认学习计划

**操作：** 进入学习计划，生成候选计划，然后确认。

**讲解：**

> 新计划首先是 candidate，不会直接覆盖当前活动计划。只有用户确认后才变为 active。今日任务只读取活动版本，因此候选计划和旧版本不会混入正在执行的任务。

### 第五步：完成今日任务

**操作：** 进入今日任务，填写实际学习分钟数并完成任务。

**讲解：**

> 完成任务后会写入学习记录并更新掌握度。重复点击和网络重试不会重复生成记录，这是通过幂等键、数据库锁和唯一约束保证的。

### 第六步：练习、错题和掌握度

**操作：** 完成一道题，可以故意答错，再打开错题本和掌握度。

**讲解：**

> 每次练习都有稳定的提交 ID。错误答案会进入错题本，并对对应知识点掌握度产生可解释的影响。错题、掌握度和后续推荐使用的是同一套真实行为数据。

### 第七步：推荐和统计

**操作：** 打开推荐中心，再打开统计页面并下载 CSV。

**讲解：**

> 推荐不是随机 AI 文案，而是根据任务、薄弱点、资料、考试日期和学习记录生成，并展示推荐理由。统计页按用户时区聚合真实学习记录，CSV 由浏览器真实下载。

### 第八步：周报与日历

**操作：** 在任务中心展示周报并下载 Markdown；在日历中同步计划任务并导出 ICS。

**讲解：**

> 周报结果会持久化，刷新后仍可以查看。日历写操作先预览再确认，并且具有幂等保护。目前是 StudyPilot 本地日历，可以通过 ICS 手动导入其他日历，Outlook 和 Google 自动同步属于后续工作。

# 老师可能提问及参考回答

## 1. 这个项目和普通大模型聊天有什么区别？

**参考回答：**

> 普通聊天工具主要解决一次性问答，而我们的项目把课程资料、计划、任务、答题、掌握度和推荐持久化，形成持续更新的学习闭环。问答还受到用户、课程、文档和版本权限约束，并提供引用，不只是自由生成文本。

## 2. 如何减少大模型幻觉？

**参考回答：**

> 后端先按照用户、课程、文档和当前版本过滤检索范围，再把检索结果编号。模型返回的引用必须来自本次允许的证据集合，证据不足时明确提示资料不足。即使后续接入外部模型，权限过滤和引用校验也由后端完成，不能交给模型自行决定。

## 3. 为什么同时使用 PostgreSQL、pgvector 和 Redis？

**参考回答：**

> PostgreSQL 保存不可丢失的权威业务数据；pgvector 是 PostgreSQL 的向量扩展，用于语义检索，并能和业务权限条件放在同一条查询中；Redis 用于可重建的短期状态、Celery 队列和任务进度。三者职责不同，出现冲突时始终以 PostgreSQL 为准。

## 4. 为什么学习计划不能直接由模型生成并覆盖？

**参考回答：**

> 学习计划会影响用户后续任务，所以我们把计划保存为结构化版本。生成后先成为 candidate，用户确认后才能变为 active。这样可以保留旧版本、支持差异比较，也避免模型输出直接覆盖用户正在执行的计划。

## 5. 如何防止用户重复点击造成重复数据？

**参考回答：**

> 前端会在提交期间锁定控件，后端还会使用幂等键、唯一约束、事务和必要的行锁。即使两个请求同时到达，数据库最终也只会产生一条有效记录。我们对任务完成、练习提交、推荐反馈和日历写入都做了对应处理。

## 6. 如何保证不同用户的数据安全？

**参考回答：**

> 前端路由守卫只能提升体验，真正的安全边界在后端。每个业务接口都会从 JWT 获取当前用户，并验证课程和资源归属。我们用 A/B/A 浏览器故事验证了课程、资料、任务、错题、反馈、周报和日历事件都不会跨用户泄漏。

## 7. 现在真的接入大模型了吗？

**参考回答：**

> 当前已经完成 Provider 接口和完整问答业务链路，但最新验收使用离线 Provider，外部 Chat 和 Embedding Provider 尚未正式接入，因此目前不评价真实模型回答质量。下一阶段会接入外部 Provider，并通过固定评测集验证检索和引用质量。

## 8. MCP 在项目中起什么作用？

**参考回答：**

> MCP 将系统能力以标准工具形式提供给 Agent。我们的 MCP Server 设计了 18 个工具。只读操作可以直接执行，写操作必须先预览并生成短时确认令牌，用户确认后再携带相同参数执行，同时记录脱敏审计日志。当前本地日历工具链已经实现，但最新验收没有启动真实 MCP Server，也尚未接入第三方日历账户。

## 9. 目前最大的不足是什么？

**参考回答：**

> 当前最大的不足不是基础页面数量，而是外部智能能力和生产化验证还没有全部完成。主要包括外部模型质量评测、第三方日历接入、最终补修后的完整矩阵回归，以及前端代码拆包和性能优化。我们已经把这些内容明确列为下一阶段任务，没有用 Mock 或静态按钮伪装成已完成功能。

## 10. 你们如何评价当前进度？

**参考回答：**

> 核心 MVP 闭环已经完成，项目已经从功能开发进入集成验收和可靠性加固阶段。现在的重点是重新跑完整回归、完成 Docker 全栈联调，并接入和评估真实外部模型，而不是继续盲目增加页面。

# 汇报时务必避免的表述

为了保证汇报准确，建议不要使用下面这些表述：

- 不要说“已经接入真实大模型”，应说“已完成 Provider 和问答链路，当前使用离线 Provider，外部模型待接入”。
- 不要说“已经自动同步 Outlook/Google”，应说“已完成本地日历和 ICS，第三方自动同步待接入”。
- 不要说“最后修改后完整测试全部重跑通过”，应说“完整验收已有通过证据，最后一次推荐锁补修完成定向桌面/移动测试，完整矩阵还需要再跑一次”。
- 不要说“所有规划功能都完成”，动态调整、延期、计时和计划历史仍是后续能力。
- 不要把 Mock 演示数据说成真实数据库结果；正式汇报尽量使用 Docker 真实模式。
