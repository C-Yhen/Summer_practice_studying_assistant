# 1. 项目简介

## 是什么

“智能学习助手”是一个面向大学生的 Web 平台。学生可以上传课程资料，基于资料进行有引用的问答，并让系统结合考试时间、空闲时间、基础水平、答题记录和知识点掌握度，生成和调整结构化学习计划。

它不是一个只会聊天的机器人，而是一个持续运行的数据闭环：

```text
用户画像 → 学习目标 → 学习计划 → 资料/题目推荐 → 学习行为
       ↑                                      ↓
个性化建议 ← 计划动态调整 ← 掌握度更新 ← 答题与任务记录
```

## 为什么做

普通大模型不了解学生私有课件，也不能可靠保存长期进度；纯课程管理系统又缺少理解资料、规划和推荐能力。本项目把 RAG、规则算法、推荐、MCP、缓存与异步任务组合成一个可验证、可解释、可持续更新的学习系统。

## 怎么实现

- Vue 3 前端负责资料上传、问答、计划确认、任务执行和可视化。
- FastAPI 负责鉴权、业务规则、数据校验和 REST API。
- PostgreSQL 保存权威业务数据，pgvector 保存资料切块向量，Redis 保存短期缓存、限流、锁和 Celery 队列。
- Celery Worker 处理解析、Embedding、批量生成、报告和日历同步等长任务。
- 自研 MCP Server 把学习数据和日历能力以受控工具暴露给 Agent。
- LLM/Embedding/Calendar/Storage 等全部通过 Provider 适配层调用，不在业务代码中写死厂商。

## 输入、处理、输出

| 输入 | 核心处理 | 输出 |
|---|---|---|
| 课程资料、考试日期、空闲时间、学习偏好 | 解析切块、画像构建、计划排程 | 可执行的每日计划 |
| 用户问题 | 权限过滤、向量检索、提示词约束、模型生成 | 带资料名、页码/章节和原文片段的答案 |
| 学习/答题/任务行为 | 统计、掌握度更新、规则判断 | 薄弱点、推荐理由、动态调整草案 |
| 用户确认 | 幂等写入、版本保存、MCP 工具调用 | 新计划版本、日历事件、审计日志 |

# 2. 项目解决的问题

| 现实问题 | 原因 | 本项目的落地方案 | 可验证结果 |
|---|---|---|---|
| 课件很多，找不到答案出处 | 普通搜索不理解语义 | 文档解析 + pgvector 语义检索 + 引用绑定 | 回答展示文档、页码、章节、原文片段 |
| 不知道先学什么 | 缺少考试约束和知识依赖 | 规则算法按重要度、薄弱度、遗忘度和前置关系排程 | 生成数据库中的每日任务，而非一段文字 |
| 计划容易失效 | 静态计划不看实际完成情况 | 监听未完成、正确率、用时和冲突，产生新版本草案 | 用户可对比差异后确认 |
| 推荐像“黑箱” | 只给结果、不说明依据 | 保存分数分解、行为依据和推荐理由 | 每条推荐都能回答“为什么适合我” |
| 长任务阻塞页面 | PDF/Embedding/批量出题耗时 | Celery 异步执行，Redis 报进度，PostgreSQL 持久化状态 | 页面可实时/轮询查看进度并重试 |
| Agent 写操作不可控 | 工具可能误创建或重复创建 | 权限、确认令牌、幂等键、调用日志 | 所有副作用有确认、有归属、有审计 |
| 第三方接口不稳定 | 厂商超时、限流、Token 失效 | Provider 适配、重试、熔断、降级、同步状态 | 更换服务不改核心业务，失败可恢复 |

# 3. 目标用户

## 用户类型

1. **基础薄弱学生**：需要补充前置知识、简单例子和分步讲解。
2. **临近考试学生**：需要倒排复习、重点题型、阶段测验和机动时间。
3. **资料较多学生**：希望同时检索 PDF、PPT、Word、TXT、Markdown。
4. **缺乏规划学生**：需要知道今天学什么、学多久、完成标准是什么。
5. **希望复盘学生**：需要错题、掌握度曲线、周报和阶段建议。
6. **教师/助教（扩展）**：需要教案、章节总结、练习题和难度分级题库。

## 用户画像输入与输出

- 输入：基础水平、目标成绩、考试时间、每日/单次时长、不可学习日期、先讲后练偏好、资料类型偏好、难度偏好、是否需要重点/易错点/推导、初测和历史行为。
- 处理：把显式偏好与学习行为聚合为课程级画像；行为事实优先，人工设置作为冷启动依据。
- 输出：`user_profiles` 的结构化快照，包括能力层级、可用时间、薄弱点、偏好、活跃度和生成依据。

# 5. 系统总体架构

## 分层架构

```text
┌──────────────── Vue 3 / TypeScript ────────────────┐
│ Axios REST：业务请求   WebSocket/轮询：任务进度     │
└──────────────────────────┬──────────────────────────┘
                           │ HTTPS / WSS
┌──────────────────────────▼──────────────────────────┐
│ FastAPI API 层：JWT、参数校验、限流、错误码          │
├─────────────────────────────────────────────────────┤
│ 服务层：课程 / 文档 / RAG / 计划 / 推荐 / 学习记录  │
│ 只在此层编排事务和权限；LLM 不直接访问数据库         │
├──────────────────────┬──────────────────────────────┤
│ Repository / Cache   │ Provider / Agent / MCP Client│
└───────┬──────────────┴───────────────┬──────────────┘
        │                              │
┌───────▼────────┐  ┌──────────────┐  ┌▼──────────────────┐
│ PostgreSQL     │  │ Redis        │  │ MCP Server / 第三方│
│ 业务表+pgvector│  │ 缓存/锁/队列 │  │ LLM/日历/存储/邮件 │
└────────────────┘  └──────┬───────┘  └───────────────────┘
                            │ Celery broker
                     ┌──────▼────────────┐
                     │ Celery Workers    │
                     │ 解析/向量/报告/同步│
                     └───────────────────┘
```

## 一次请求经过哪些层

以“对数据库课件提问”为例：

1. 前端发送课程、会话、问题、模式和可选文档范围。
2. API 层验证 JWT、课程成员身份、参数长度和限流。
3. Chat Service 计算带版本的缓存键；命中时校验引用版本仍有效。
4. 未命中时 RAG Service 生成向量，Repository 在 pgvector 中按课程、文档归属和当前版本过滤。
5. 将检索片段编号为 `[S1]`、`[S2]`，LLM Provider 返回结构化 JSON。
6. 后端验证引用 ID 必须来自本次检索集合，严格模式下无充分证据则拒答。
7. 在一个业务事务中保存用户消息、助手消息和引用快照，之后写缓存。
8. 返回前端展示；失败时返回统一 `request_id` 便于排查。

## 关键边界

- 前端不直接访问 Redis、数据库、Celery 或第三方密钥。
- API 层不写复杂 SQL；Repository 封装数据访问，Service 封装业务事务。
- MCP Server 复用业务 Service 或调用内部受保护 API，不能绕过权限直接改表。
- Worker 接收 ID 和版本，不接收不可控的大对象；每一步可重入。
- 文件二进制放对象存储/本地开发存储，数据库只存元数据和路径。

## 部署建议

本地答辩使用 Docker Compose：`frontend + api + mcp-server + celery-worker + postgres(pgvector) + redis + minio(可选)`。生产化再拆反向代理、监控和多 Worker；实训期不要过早做 Kubernetes。

# 6. PostgreSQL、pgvector 和 Redis 的职责划分

## 是什么与为什么

PostgreSQL 是权威事实库；pgvector 是 PostgreSQL 内的向量扩展，不是另一套业务数据库；Redis 是可丢失、可重建的高速层。这样既能通过事务保持课程/计划/行为一致，又能在同一 SQL 中完成向量相似度和权限/版本过滤。

| 组件 | 保存什么 | 不应保存什么 | 一致性策略 |
|---|---|---|---|
| PostgreSQL | 用户、课程、文档版本、计划版本、任务、答题、掌握度、推荐、聊天、MCP/异步日志、日历映射 | 原文件二进制、大量临时进度 | 唯一事实来源，事务提交成功才算写入成功 |
| pgvector | `document_chunks.embedding` 及切块元数据 | 用户密码、任务队列、会话 | 与切块同事务写入；检索只查当前有效版本和未软删记录 |
| Redis | QA/检索/画像/计划缓存，会话状态，限流计数，锁，Celery broker，任务进度 | 唯一业务记录、永久聊天、唯一任务状态 | Cache-Aside；DB 先写、缓存后删；缓存丢失可从 DB 恢复 |

## 典型数据归属

```text
上传文件：对象存储保存原件
          PostgreSQL.documents/document_versions 保存元数据
          PostgreSQL.document_chunks.embedding 保存向量

长任务：  PostgreSQL.async_tasks 保存关键状态和最终结果
          Redis job:{task_id} 保存高频进度
          Redis/Celery 队列保存待执行消息

学习计划：PostgreSQL 保存所有版本和任务
          Redis 仅缓存当前已确认版本的只读视图
```

## 故障时如何处理

- Redis 不可用：降级查 PostgreSQL；限制昂贵 RAG 请求，任务提交视 broker 状态返回 `503`，不得假装已入队。
- pgvector 检索慢：先检查过滤索引、切块量和 `EXPLAIN ANALYZE`；小数据可精确检索，数据量大再启用 HNSW。
- PostgreSQL 不可用：停止所有写操作；不从缓存返回会被误解为“刚写成功”的结果。
- Redis 与 DB 状态不一致：DB 为准，通过版本号拒绝旧缓存并重建。
