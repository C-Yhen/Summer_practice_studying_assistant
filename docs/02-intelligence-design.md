# 7. RAG 实现流程

## 是什么

RAG（检索增强生成）先从学生自己的资料中找证据，再让大模型依据证据回答。它解决“模型不知道私有课件”和“回答没有出处”两个问题。这里的核心不是让模型记住文件，而是让每次回答都能追溯到特定文档版本、页码和文本块。

## 输入、处理、输出

| 阶段 | 输入 | 处理 | 输出 |
|---|---|---|---|
| 建库 | 原文件、课程 ID、文档版本 | 解析、清洗、切块、Embedding、写入 pgvector | 可检索切块及页码/章节元数据 |
| 查询 | 用户、课程、问题、模式、文档范围、会话上下文 | 权限校验、缓存、检索、重排、提示词、模型调用、引用校验 | 结构化答案、引用、置信提示 |
| 反馈 | 点赞/踩、引用点击、追问 | 保存行为、统计失败样本 | RAG 评测数据和改进依据 |

## 文档建库流程

```text
上传 → 文件安全校验 → 保存原件 → documents/document_versions 记录
    → Celery 解析 → 保留页码/章节 → 清洗 → 切块
    → 批量 Embedding → 批量写 document_chunks → 抽样校验
    → 原子切换 current_version → 清理旧版本缓存 → ready
```

### 解析和切块规则

1. 允许 PDF、PPT/PPTX、DOC/DOCX、TXT、Markdown；MVP 先稳定支持 PDF。
2. 校验扩展名、MIME、文件头、大小和课程归属，拒绝宏、可执行内容和路径穿越。
3. PDF 按页提取，PPT 按幻灯片，Word/Markdown 按标题层级；页码或章节作为元数据保留。
4. 删除连续空白、重复页眉页脚和不可见字符，但不改写公式和原文事实。
5. 推荐初始切块为 400～700 中文字符、重叠 80～120 字符；优先在标题/段落边界切，不要从一句话中间切开。
6. 每块保存 `document_id + document_version + chunk_index + page_number + chapter_name + content_hash`；相同内容哈希可跳过重复 Embedding。
7. Embedding 维度由模型决定，例如 1024；数据库迁移中的 `VECTOR(n)` 必须与 Provider 返回维度一致，不能运行时随意切模型。
8. 新版本全部切块成功后才切换为当前版本；失败时旧版本仍可服务。

## 问答流程

1. **鉴权**：JWT 获取 `user_id`，确认用户是课程成员，指定文档属于该课程且未删除。
2. **规范化**：去除首尾空格，限制问题长度；把课程、当前版本集合、模式、Top-K 和问题规范化后计算哈希。
3. **查缓存**：读取 `qa:{user}:{course}:{version_fingerprint}:{question_hash}`；缓存值中的版本指纹必须仍等于数据库当前版本指纹。
4. **构造查询**：单轮问题直接向量化；多轮问题可由小模型把代词改写成独立问题，但保存原问题，改写失败则使用原问题。
5. **候选召回**：pgvector 语义检索 Top 20；可选 PostgreSQL 全文检索 Top 20；必须在 SQL 中过滤课程、允许文档、当前版本和软删标记。
6. **合并去重**：可用 Reciprocal Rank Fusion（RRF）合并向量和关键词名次；MVP 可先只做向量检索。
7. **重排**：可选 reranker 对候选重新打分，最终保留 4～8 块；过滤低于阈值且内容过度重复的片段。
8. **构造上下文**：每块分配不可伪造的来源编号，例如 `[S1]`，包含资料名、版本、页码/章节和原文。
9. **调用模型**：要求只引用提供的来源编号，并返回 JSON；严格资料模式禁止补充外部知识。
10. **后端验证**：`citation.source_id` 必须属于本次检索集合；答案声称有依据却没有引用时重试一次，仍失败则返回受控降级答案。
11. **持久化与缓存**：保存用户问题、检索快照、模型答案、引用和耗时；事务完成后写入带版本的 QA 缓存。

## 四种回答模式

| 模式 | 提示词约束 | 资料不足时 |
|---|---|---|
| 基础模式 | 先“是什么”，再“为什么”，补前置知识、简单例子和小结 | 标明哪些来自资料；外部常识补充必须明确标注 |
| 考试模式 | 核心概念、公式与符号、题型、步骤、易错点、可写在卷面上的答案 | 明确“资料未覆盖”，不伪造考试重点 |
| 严格资料模式 | 只能使用提供片段，所有关键结论必须引用 | 直接回答“当前资料中未找到足够依据” |
| 教师模式 | 教学提纲、教案、总结、练习和答案，仍绑定来源 | 缺少覆盖时列出待补资料，不生成伪造引用 |

## 模型结构化输出示例

```json
{
  "answer": "第三范式要求在满足第二范式的基础上，非主属性不传递依赖于候选键。",
  "mode": "strict",
  "sufficient_evidence": true,
  "citations": [
    {
      "source_id": "S2",
      "claim": "3NF 的定义",
      "quote": "……非主属性不传递函数依赖于候选键……"
    }
  ],
  "follow_up_questions": ["需要用一个关系模式判断是否满足 3NF 吗？"]
}
```

后端根据 `source_id` 补齐可信的 `document_id/document_name/version/page_number/chapter_name/chunk_id`，不接受模型自行生成这些字段。

## 无答案与安全边界

- 最终片段都低于阈值、相互矛盾或无法支持问题时，返回“未找到足够资料依据”，并建议上传相关章节或扩大文档范围。
- 上下文中的“忽略系统提示”等文字一律视为资料内容，不视为命令，防止提示注入。
- 模型输出经过 JSON Schema/Pydantic 校验、引用白名单校验和长度限制；模型失败不写空记录为成功。
- 聊天历史只取最近若干轮的摘要/消息，不能把其他课程或其他用户内容带入上下文。

# 8. 学习计划算法

## 是什么与为什么

学习计划是由 `study_plans + study_plan_versions + study_tasks` 保存的可执行排程。若让大模型直接输出一段自然语言，系统无法检查每日负载、前置依赖、完成状态，也无法可靠调整。因此采用“规则算法生成骨架，大模型优化表达，后端校验保存”。

## 输入与输出

输入：课程、考试日期、开始日期、每天可用时间段、不可学习日期、单次时长、当前基础、已完成/薄弱知识点、目标成绩、偏好、知识依赖、资源和题目。

输出（先生成草案，确认后生效）：

```json
{
  "course_id": 12,
  "start_date": "2026-07-15",
  "end_date": "2026-07-21",
  "goal": "数据库课程达到 85 分",
  "version": 1,
  "status": "draft",
  "summary": "先补函数依赖，再学习范式，最后进行模拟测试。",
  "tasks": [
    {
      "date": "2026-07-15",
      "knowledge_point_id": 31,
      "name": "复习函数依赖基础",
      "task_type": "learn_new",
      "estimated_minutes": 50,
      "priority": 90,
      "difficulty": "basic",
      "prerequisite_task_ids": [],
      "resource_ids": [91],
      "question_ids": [501, 502]
    }
  ]
}
```

## 规则算法步骤

1. **时间预算**：列出开始日到考试日前一天的可学习日期，扣除不可学习日期和已有日历冲突；每天保留 10%～20% 机动时间。
2. **学习对象**：获取课程知识点、重要度、预计时长、前置关系、掌握度、最近复习时间、题目和资源覆盖情况。
3. **依赖检查**：对知识依赖图进行拓扑排序；出现环时标记数据错误并停止自动排程，不能随便打乱。
4. **优先级计算**：把各项归一化到 `[0,1]`，避免简单相乘出现全零：

```text
priority = 100 × (
  0.30 × exam_importance
  + 0.25 × weakness
  + 0.15 × forgetting
  + 0.15 × prerequisite_impact
  + 0.15 × urgency
)

weakness = 1 - mastery_score
forgetting = clamp(days_since_review / review_interval, 0, 1)
urgency = clamp(1 - days_left / configured_horizon, 0, 1)
```

附件中的乘法公式可作为解释性概念，但工程上加权和更稳定；若必须使用乘法，应给每项设置下限（如 0.1）。

5. **生成任务**：每个未掌握知识点通常生成“学习/阅读 → 小练习 → 间隔复习”；重要章节增加阶段测试，考前保留综合复习和模拟考试。
6. **排入日期**：按优先级从高到低，先满足前置任务，再寻找剩余时长足够的日期；单个任务超过单次时长时拆分。
7. **防过载**：每日预计时间不得超过可用时间的 90%；高难任务之间插入低负担复习；同类任务连续数量设上限。
8. **间隔复习**：可用第 1、3、7 天作为简单间隔；考试更近时压缩为第 1、2、4 天。
9. **可行性检查**：若总需求时长大于预算，保留高重要/薄弱/前置内容，降级低优先级内容并输出风险，而不是生成无法完成的计划。
10. **LLM 润色**：仅传入脱敏的结构化骨架，请模型优化任务名称、说明、安排理由和总结，不允许改变 ID、日期、时长总量和依赖约束。
11. **Pydantic 校验**：检查日期范围、分钟数、枚举、ID 归属、前置顺序、每日负载和资源存在性。
12. **先草案后确认**：保存 `draft` 版本并返回差异/风险；用户确认后用事务切换为 `active`。

## 核心伪代码

```python
def build_plan(context):
    days = build_available_days(context.calendar, reserve_ratio=0.15)
    points = topological_sort(context.knowledge_points)
    for point in points:
        point.priority = priority_score(point, context)

    candidates = []
    for point in sorted(points, key=lambda x: x.priority, reverse=True):
        candidates += make_learning_review_test_tasks(point, context.preferences)

    schedule, unscheduled = place_with_dependencies(candidates, days)
    assert_no_daily_overload(schedule)
    skeleton = to_plan_schema(schedule, unscheduled)
    wording = llm_provider.rewrite_plan(skeleton)  # 只优化文字
    return validate_and_merge_wording(skeleton, wording)
```

## 异常和边界

- 考试日期已过：返回 `PLAN_DATE_INVALID`，提示修改考试日期。
- 当天没有可用时间：不排任务，并在后续日期重新分配。
- 时间严重不足：输出可完成的“保底计划”和未安排列表，不能偷偷把每日时间加大。
- 没有初测：按用户自报基础和课程重要度冷启动，并降低掌握度置信度。
- LLM 不可用：规则骨架仍可生成，只使用模板文字。

# 9. 动态计划调整算法

## 是什么与为什么

动态调整不是直接修改原任务，而是根据新事实生成一个候选版本，展示差异并等待用户确认。保留旧版本能解释“为什么改”和支持审计/回滚，也避免系统在用户不知情时改变日程。

## 触发器

| 触发条件 | 建议阈值 | 初步动作 |
|---|---:|---|
| 连续未完成 | 2 天 | 降低每日负载、迁移高优先级未完成任务 |
| 同知识点连续低正确率 | 最近 3 次均低于 60% | 增加前置学习、基础资料和错题复习 |
| 高掌握 | 正确率超过 90% 且样本不少于 5 | 减少基础题，保留间隔复习 |
| 用时明显偏低 | 实际/预计低于 0.6，连续 3 项 | 检查是否跳过；确认后可提高强度 |
| 用时明显偏高 | 实际/预计高于 1.5，连续 3 项 | 拆分任务、降低难度、补前置知识 |
| 空闲时间改变 | 用户更新或日历冲突 | 重新计算每日容量 |
| 考试变化 | 日期提前/延后 | 重算紧迫度和阶段任务 |
| 用户主动调整 | 随时 | 以用户约束为最高优先级 |

阈值必须配置化并记录命中依据，不能只让大模型“感觉需要调整”。

## 输入、处理、输出

- 输入：当前已确认版本、未来未完成任务、触发证据、最新可用时间、掌握度、知识依赖、考试日期、日历冲突。
- 处理：冻结已完成和正在执行的任务；重算优先级；把未完成任务、薄弱点补强任务和阶段测试重新装入剩余容量；生成差异。
- 输出：`draft_adjustment` 新版本，包括原因、证据、增加/删除/推迟/修改任务、总时长变化、未覆盖内容和风险。

## 调整步骤

1. 事件写入后运行规则引擎，得到结构化 `trigger_evidence`。
2. 使用分布式锁 `lock:plan:{user_id}:{course_id}` 防止两个调整同时生成。
3. 读取当前 `active` 版本并冻结过去日期、已完成任务及用户锁定任务。
4. 重新计算未来任务优先级和剩余时间容量。
5. 优先处理高优先级未完成任务；过长任务拆分；低价值内容放入 `removed`，但保留删除原因。
6. 为薄弱知识点插入学习/错题复习；对已掌握点减少重复基础练习；临考增加模拟题。
7. 运行前置关系、时间冲突、每日负载和总容量检查。
8. 创建新 `study_plan_versions(status='draft')` 和任务快照，旧版本保持不变。
9. 返回调整前后差异和风险，并签发短期一次性确认令牌。
10. 用户确认时再次验证当前版本号未变化；事务内将旧版本标为 `superseded`、新版本标为 `active`。
11. 提交后删除相关计划缓存；如需同步日历，再单独展示事件变更并确认，不能把两个确认偷偷合并。

## 差异示例

```json
{
  "base_version": 2,
  "candidate_version": 3,
  "reasons": [
    {"rule": "LOW_ACCURACY_3_TIMES", "evidence": {"knowledge_point_id": 31, "accuracy": 0.52}}
  ],
  "added": [{"task_temp_id": "new-1", "name": "函数依赖错题复习", "minutes": 30}],
  "removed": [{"task_id": 808, "reason": "低优先级且剩余时间不足"}],
  "rescheduled": [{"task_id": 802, "from": "2026-07-16", "to": "2026-07-17"}],
  "minutes_change": 20,
  "risks": ["事务调度综合题仅安排一次练习"],
  "requires_confirmation": true,
  "confirmation_token": "short-lived-opaque-token"
}
```

拒绝或令牌过期时保持原版本；确认接口必须接收幂等键，重复提交返回同一结果。

# 10. 推荐系统设计

## 是什么与为什么

第一阶段采用“规则 + 内容标签”的可解释推荐，不需要训练深度模型。它能在数据少时工作，分数可追溯，也适合 8 周项目。推荐对象包括资料、章节、知识点、练习、错题、每日任务、复习内容和学习顺序。

## 输入、处理、输出

| 输入 | 处理 | 输出 |
|---|---|---|
| 用户画像、知识掌握度、计划阶段、考试剩余时间 | 找出当前学习意图和待补知识点 | 推荐上下文 |
| 资源/题目的课程、知识点、难度、类型、用时、质量、前置标签 | 权限过滤、候选召回、规则打分、去重和多样性控制 | 排序后的候选 |
| 点击、查看时长、完成、跳过、收藏、评分、正确率变化 | 写行为事实并更新统计 | 评价指标和后续权重依据 |

## 候选召回与过滤

1. 只召回用户有权访问、未删除、状态可用且属于目标课程的对象。
2. 从“当前任务知识点、薄弱点、前置点、错题点、考前重点”分别召回，避免单一路径漏掉候选。
3. 过滤已明确跳过且仍在冷却期、难度跨度过大、预计用时超过可用时间、前置知识未满足的资源。
4. 同一文档章节/同类题目设置上限，保证结果有一定多样性。

## 打分公式

每个子分统一归一化到 `[0,1]`：

```text
score = 100 × (
  0.30 × knowledge_match
  + 0.25 × weakness
  + 0.20 × difficulty_match
  + 0.10 × preference_match
  + 0.10 × resource_quality
  + 0.05 × time_match
)
```

- `knowledge_match`：候选知识点与当前任务/薄弱点的匹配程度。
- `weakness`：`1 - mastery_score`，但低置信度时向 0.5 收缩。
- `difficulty_match`：候选难度与用户能力相近得分最高，允许略高一级。
- `preference_match`：先讲后练、资料类型和学习方式是否匹配。
- `resource_quality`：人工评分、完成率、反馈质量；样本少时使用课程平均，防止小样本极端值。
- `time_match`：预计用时是否落入当前空闲时段。

可加入业务约束而不偷偷改变解释：已学过但需复习可乘 `1.05`，近期连续跳过可乘 `0.8`；这些调整也要存入分数明细。

## 可解释理由

系统先用事实模板生成理由，再可选交给 LLM 润色。LLM 只能改表达，不能添加不存在的正确率或因果关系。

```json
{
  "item_type": "resource",
  "item_id": 91,
  "score": 86.5,
  "score_breakdown": {
    "knowledge_match": 1.0,
    "weakness": 0.48,
    "difficulty_match": 0.9,
    "preference_match": 1.0,
    "resource_quality": 0.82,
    "time_match": 1.0
  },
  "reason_facts": [
    "函数依赖最近正确率为 52%",
    "该资料为基础难度",
    "函数依赖是 BCNF 的前置知识",
    "预计 25 分钟，适合当前 30 分钟空闲时段"
  ],
  "reason": "你在函数依赖上的最近正确率为 52%，这份基础资料又是学习 BCNF 的前置内容，预计 25 分钟可在当前时段完成。"
}
```

每次曝光写入 `recommendation_records`，保存当时的候选类型、对象 ID、算法版本、总分、分项分、理由事实、排序位置和上下文快照。否则后续无法判断推荐是否有效。

## 新用户冷启动

新用户没有行为记录时，按以下顺序使用数据：初始测试 → 自报基础 → 目标成绩与考试时间 → 课程重要度 → 资源难度/质量。若没有初测，推荐一份短诊断题、基础导读和高覆盖资料，并明确理由是“尚无足够行为数据”。至少完成 5～10 次行为后再逐渐提高个性化行为权重。

## 行为反馈与掌握度

- 点击/查看只是弱正反馈，收藏/完成/评分是中等反馈，推荐后答题正确率或掌握度提升是强反馈。
- “没点击”不等于不喜欢，只有明确跳过/差评才作为负反馈。
- 推荐系统只能读取掌握度，掌握度更新由独立服务根据答题和复习证据完成，避免推荐本身制造“已掌握”。
- 展示指标包括点击率、完成率、推荐后正确率变化、理由认可率和冷启动完成率；MVP 不用在线自动调权，先离线分析、人工版本化权重。

## 个性化学习建议的三层实现

学习建议与推荐不同：推荐回答“接下来给你什么内容”，建议回答“根据最近数据，你应怎样调整学习方式”。它仍遵循“数据判断、规则约束、模型表达”。

1. **规则层**输入连续学习时长、最近三次正确率、连续未完成天数、考试剩余日和题型耗时；处理固定阈值，例如连续超过 90 分钟建议休息、最近三次正确率都低于 60% 建议重学、考试剩 3 天进入综合复习；输出 `rule_code + evidence + action`。
2. **分析层**输入时长、完成率、正确率、掌握度、错题、计划偏差、推荐反馈和趋势；处理 7/30 天聚合、同一用户前后对比和异常值过滤；输出结构化趋势和风险，不用小样本下的单次波动下结论。
3. **表达层**只接收前两层 JSON，让 LLM 改写成基础一般学生能看懂的建议；后端校验建议中出现的数值和知识点必须来自输入事实。模型不可自行判断“你函数依赖很弱”。

```json
{
  "advice_type": "relearn_knowledge_point",
  "severity": "medium",
  "evidence": {
    "knowledge_point_id": 31,
    "knowledge_point_name": "函数依赖",
    "recent_attempts": 3,
    "accuracy": 0.52
  },
  "actions": ["先阅读基础讲解", "重做两道错题", "明天进行一次短测"],
  "message": "你最近 3 次函数依赖练习的正确率为 52%，建议先回看基础讲解，再重做错题。"
}
```

输入证据、规则版本、生成时间和展示状态应持久化到通知/报告或建议记录快照；MVP 若不新增建议表，可将每日建议作为 `notifications.data_json` 保存，但不要只存在前端内存中。
